# When OpenClaw Meets Hospital: 医疗智能体操作系统深度解析

## 论文基本信息

- **标题**: When OpenClaw Meets Hospital: Toward an Agentic Operating System for Dynamic Clinical Workflows
- **作者**: Wenxian Yang, Hanzheng Qiu, Bangqun Zhang, Rongshan Yu 等
- **机构**: 清华大学、厦门大学、新加坡国立大学、北京清华长庚医院
- **发表时间**: 2026年3月

- **引用量**: 新发表论文

## 研究背景与动机

### 医疗领域的AI机遇

医疗健康是人工智能最具影响力也最具挑战性的应用领域之一。大型语言模型(LLM)的出现为医疗AI带来了新的可能性：

**临床文档处理**
医生每天花费大量时间撰写病历、出院小结等文档。LLM可以自动生成文档草稿、提取关键信息、辅助编码，显著减轻医护人员的文书负担。

**护理协调**
复杂的医疗流程涉及多科室协作、转诊安排、随访计划等。LLM Agent可以自动化协调流程，减少沟通成本和人为错误。

**临床决策支持**
基于患者病史、检查结果和最新医学指南，LLM可以提供个性化诊断建议、用药提醒、风险预警等决策支持。

### 医疗场景的特殊挑战

尽管前景广阔，LLM在医疗场景的部署面临独特挑战：

**安全风险**

医疗决策直接关系到患者生命健康。LLM可能产生"幻觉"——生成看似合理但实际错误的医学信息，这在医疗场景是不可接受的。如何确保Agent输出的准确性和安全性是首要问题。

**透明度不足**

医疗决策需要可解释性。医生需要理解AI的建议依据，才能做出最终判断。然而，LLM的黑盒特性使其决策过程难以审计，这限制了其在临床中的信任度和采纳率。

**长期上下文管理**

患者可能经历多次就诊、不同科室、长期治疗。完整的患者画像需要整合分散在各处的信息，跨越时间跨度可能达数年。现有系统难以有效管理这种纵向(longitudinal)临床上下文。

**系统集成复杂性**

医院信息系统(HIS)、电子病历(EMR)、实验室信息系统(LIS)等众多系统需要整合。LLM Agent需要与这些遗留系统交互，同时保持安全边界。

### OpenClaw：开源智能体框架

OpenClaw是一个开源的LLM Agent编排框架，提供了工具调用、记忆管理、安全沙箱等功能。本研究选择OpenClaw作为基础平台，探索其在医疗场景的系统化应用。

### 研究动机总结

本研究旨在回答一个核心问题：**如何将LLM Agent系统化地适配到医院环境，在保证安全、透明、可审计的前提下，支持动态临床工作流？**

## 核心方法详解

### AOS-H架构概览

论文提出了Agentic Operating System for Hospitals (AOS-H)架构，包含四个核心组件：

```
┌─────────────────────────────────────────────────────────┐
│                    AOS-H 架构                           │
├─────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ 受限执行环境 │  │文档中心交互  │  │ 页面索引记忆 │  │
│  │ (Restricted  │  │ 模型         │  │ 架构         │  │
│  │ Environment) │  │              │  │              │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                         │
│  ┌──────────────────────────────────────────────────┐  │
│  │          可组合医疗技能库                          │  │
│  │   (Curated Medical Skills Library)               │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 组件一：受限执行环境

**设计理念**

借鉴多用户操作系统(Multi-user OS)的设计原则，为每个Agent实例提供隔离的执行环境：

**权限控制**
- 最小权限原则：Agent只能访问完成任务所必需的资源
- 角色基础的访问控制(RBAC)：根据Agent角色（医生助手、护士助手等）设置不同权限
- 审计日志：所有Agent行为都被记录，支持事后追溯

**资源隔离**
- 内存隔离：不同Agent实例的内存空间相互独立
- 文件系统隔离：Agent只能访问授权的文档目录
- 网络隔离：限制Agent的外部访问范围

**执行边界**
- 超时控制：防止Agent陷入无限循环
- 输出过滤：检查并过滤潜在有害输出
- 人工确认：关键操作需要人工确认

**实现细节**

基于OpenClaw的沙箱机制实现：
- 容器化部署：每个Agent运行在独立容器中
- 系统调用过滤：限制可执行的系统调用
- 资源配额：CPU、内存、存储的限制

### 组件二：文档中心交互模型

**核心思想**

以文档为中心组织Agent之间的交互，而非传统的消息传递模式：

**文档即接口**

医疗场景中，病历、医嘱、报告等文档是信息的载体。AOS-H将文档作为Agent交互的主要接口：
- 患者Agent负责维护患者主文档
- 临床医生Agent读取和更新文档
- 所有交互都有文档记录，支持审计

**双向Agent架构**

每个临床场景涉及两类Agent：

1. **患者Agent (Patient Agent)**
   - 维护患者病史、检查结果、用药记录
   - 整合来自不同系统的信息
   - 提供患者视角的信息查询

2. **临床医生Agent (Clinician Agent)**
   - 访问患者Agent获取信息
   - 生成临床文档和建议
   - 与医生协作完成决策

**文档版本控制**

- 所有文档变更都有版本历史
- 支持回滚和差异对比
- 关键变更需要数字签名

### 组件三：页面索引记忆架构

**问题背景**

患者记录可能跨越数年时间、多次就诊，累积数千页文档。如何让LLM Agent高效检索相关上下文是核心挑战。

**传统方法的局限**

**全量检索**：将所有文档作为上下文输入LLM
- 问题：超出LLM上下文窗口限制
- 成本：token消耗巨大

**向量检索(RAG)**：基于语义相似度检索
- 问题：可能遗漏关键但语义不相似的信息
- 精度：检索结果可能包含大量噪声

**论文创新：Manifest-Guided检索**

AOS-H提出了分层导航策略：

**层级结构**

```
患者记录 Manifest
├── 就诊记录索引
│   ├── 2026-03-15 门诊
│   ├── 2026-02-20 急诊
│   └── 2025-12-10 住院
├── 检查报告索引
│   ├── 实验室检查
│   ├── 影像检查
│   └── 病理报告
└── 用药记录索引
    ├── 当前用药
    └── 历史用药
```

**检索过程**

1. **Manifest解析**：LLM首先解析Manifest，理解记录的整体结构
2. **层级导航**：根据查询需求，逐层定位相关记录
3. **精确提取**：从定位的文档中提取关键信息

**优势**

- **高效**：避免扫描全部文档
- **精确**：通过结构化导航减少误检
- **可解释**：检索路径清晰可见

### 组件四：可组合医疗技能库

**设计理念**

将医疗场景的常用能力封装为独立的技能模块：
- 每个技能专注于单一功能
- 技能可自由组合完成复杂任务
- 所有技能经过严格审核

**技能类型**

**信息提取技能**
- 实验室结果解析：从检验报告中提取关键指标
- 影像报告结构化：将自由文本报告转为结构化数据
- 用药信息提取：从医嘱中提取药物、剂量、频次

**文档生成技能**
- 出院小结生成：基于住院记录自动生成出院小结
- 病历摘要生成：生成患者病史概述
- 转诊信生成：生成转诊所需的医学信函

**决策支持技能**
- 药物相互作用检查：检查多药物联用的潜在风险
- 异常值预警：识别异常检验结果并提示
- 诊断编码建议：根据病历内容推荐ICD编码

**技能审核机制**

- 医学专家评审：每个技能需通过临床专家审核
- 测试集验证：在标注数据集上验证准确性
- 持续监控：部署后持续监控技能表现

## 实验设计与结果

### 实验设置

**数据集**

基于MIMIC-IV v2.2数据集构建评估基准：
- 100个去标识化患者记录
- 300个临床查询，分为三个难度层级

**难度分层**

| 层级 | 描述 | 查询数量 |
|------|------|---------|
| Tier 1 | 单次就诊内的简单查询 | 100 |
| Tier 2 | 跨多次就诊的相关查询 | 100 |
| Tier 3 | 复杂纵向查询，跨越长时间和多科室 | 100 |

**评估指标**

- **召回率(Recall)**：正确检索的相关信息占所有相关信息的比例
- **精确率(Precision)**：检索到的信息中正确的比例
- **检索文档数量**：为回答查询需要检索的文档数量

**对比方法**

- **Metadata-filtered RAG**：基于元数据过滤的向量检索
- **Manifest-guided retrieval**：论文提出的方法

### 主要结果

**整体性能**

| 方法 | 召回率 | 精确率 | 平均检索文档数 |
|------|--------|--------|---------------|
| Metadata-filtered RAG | 0.876 | 0.352 | 45.2 |
| Manifest-guided | 0.877 | 0.779 | 12.8 |

Manifest-guided检索在召回率相当的情况下，精确率是基线方法的2.2倍，检索文档数量减少约70%。

**分层分析**

| 难度层级 | RAG召回率 | Manifest召回率 | Manifest提升 |
|---------|----------|---------------|-------------|
| Tier 1 | 0.921 | 0.918 | -0.3% |
| Tier 2 | 0.891 | 0.895 | +0.4% |
| Tier 3 | 0.701 | 0.846 | +21% |

关键发现：**在复杂纵向查询(Tier 3)上，Manifest-guided方法的召回率提高21%**。这表明LLM引导的分层导航在处理跨时间、跨科室的复杂查询时具有显著优势。

**定性分析**

论文提供了具体案例说明Manifest-guided检索的优势：

**案例：跨科室诊断关联**

查询："患者2025年的心脏问题是否与肾脏病史相关？"

**RAG方法**：
- 检索了50+文档，包含大量噪声
- 召回率0.65，遗漏了部分关键记录

**Manifest-guided方法**：
- 通过Manifest定位到2025年的心内科和肾内科就诊记录
- 仅检索12个相关文档
- 召回率0.92，完整捕获了诊断关联

## 关键创新与贡献

### 系统级设计

**医院智能体操作系统**

首次系统化提出面向医院的LLM Agent操作系统架构，将安全、透明、可审计性作为一等公民纳入设计：
- 不是简单的API封装，而是完整的系统级解决方案
- 不是单点创新，而是多组件协同设计

**多用户OS隐喻**

借鉴操作系统设计原则，为医疗Agent提供了成熟的隔离和权限管理机制，这是将LLM Agent从"玩具"变为"生产系统"的关键一步。

### 技术创新

**Manifest-guided检索**

开创性地将LLM用于文档检索的导航而非直接生成，实现了：
- 更高的检索效率（减少70%文档检索量）
- 更高的精确率（2.2倍提升）
- 更好的可解释性（导航路径可见）

**文档中心交互**

以文档为Agent交互接口的设计，天然契合医疗场景的工作流特点，同时提供了完整的审计轨迹。

### 实践贡献

**开源基础**

基于OpenClaw开源框架实现，降低了医院部署的门槛：
- 医院可以自主部署和定制
- 社区可以贡献和改进
- 促进医疗AI的民主化

**可评估基准**

构建了基于MIMIC-IV的评估基准，为后续研究提供了标准化测试平台。

## 后续影响与演进

### 医疗AI的新范式

AOS-H为医疗AI的落地提供了新思路：

**从单点到系统**

传统医疗AI研究多关注单一任务（如影像诊断、报告生成）。AOS-H展示了系统级设计的价值，为构建完整的医疗AI助手奠定了基础。

**从黑盒到透明**

强调可解释性和可审计性，回应了医疗场景对AI信任的核心诉求。这可能推动监管机构对医疗AI的接受度提升。

### 技术演进方向

**技能生态建设**

未来可以构建更丰富的医疗技能库：
- 覆盖更多科室和场景
- 支持医院定制开发
- 建立技能认证标准

**多模态整合**

整合影像、语音等多模态信息：
- 医学影像解读
- 语音病历录入
- 多模态诊断支持

**联邦学习**

支持跨医院的联合训练：
- 保护患者隐私
- 提升模型泛化能力
- 促进医学知识共享

## 局限性与改进方向

### 技术局限

**Manifest构建成本**

Manifest需要手动或半自动构建，增加了初始部署成本。

**改进方向**：
- 自动化Manifest生成
- 从电子病历结构中自动提取

**技能覆盖范围**

目前技能库覆盖的场景有限，复杂罕见病场景支持不足。

**改进方向**：
- 扩展技能覆盖范围
- 支持用户自定义技能

### 实践挑战

**系统集成**

与医院现有信息系统的集成仍然复杂，需要大量定制工作。

**改进方向**：
- 标准化接口设计
- 提供更多集成模板

**临床验证**

论文主要在MIMIC数据集上验证，真实临床环境的验证仍在进行中。

**改进方向**：
- 开展多中心临床试验
- 建立长期效果追踪机制

## 总结与启示

### 核心价值

AOS-H的贡献不仅在于技术创新，更在于展示了如何将LLM Agent系统化地适配到高要求的专业领域：

**安全优先**

在医疗场景，安全不是可选项，而是必备项。AOS-H通过受限执行环境、权限控制、审计日志等机制，将安全融入架构设计。

**透明可解释**

通过文档中心交互、Manifest导航等技术，使Agent的行为可追溯、可审计，建立了医患信任的基础。

**效率与质量平衡**

Manifest-guided检索在提升效率的同时提高了精确率，展示了"正确方法可以兼得"的可能性。

### 对研究者的启示

**系统思维**

解决真实世界问题需要系统级思考，而非单点优化。AOS-H的成功源于多组件协同设计。

**领域知识**

深入理解医疗场景的特殊需求，才能设计出真正有用的系统。跨学科合作至关重要。

**开源精神**

基于开源框架构建，既降低了部署门槛，也促进了社区发展。

### 对实践者的启示

**渐进式部署**

可以从单一科室或场景开始试点，逐步扩展：

1. 先在低风险场景（如文档生成）验证
2. 积累信任后扩展到决策支持
3. 最终实现全院级部署

**人机协作**

设计为增强人类能力而非替代人类：
- Agent处理常规任务
- 医生专注于复杂决策
- 人工审核关键输出

**持续改进**

建立反馈机制：
- 收集使用中的问题
- 定期更新技能库
- 优化检索策略

### 未来展望

AOS-H为医疗智能体开辟了新的研究方向：

**从助手到伙伴**

未来的医疗Agent不仅是工具，更是医生的智能伙伴，能够理解上下文、预测需求、主动提供建议。

**个性化医疗**

结合患者基因组、生活方式等数据，提供真正个性化的诊疗建议。

**预防医学**

从治疗转向预防，通过分析长期数据预测健康风险，实现早期干预。

AOS-H的故事告诉我们：AI在医疗领域的应用，需要的不仅是强大的模型，更是深思熟虑的系统设计。安全、透明、可审计性应该是一等公民，而非事后补救。只有这样，AI才能真正成为医疗领域的可靠助手。
